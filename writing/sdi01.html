Distributed Systems Design - Design a URL shortening service


Let's design a URL shortening service like bit.ly or tinyurl.com. This service will provide short aliases redirecting to long URLs.


1. What is the purpose of shortened URL?

Shortened URLs save a lot of space when printed, displayed, or messaged. They're also easier to remember.


2. Requirements and goals

The URL shortening service should meet the following requirements:

  • Given a URL the service should generate a shorter and unique alias.
  • If a user clicks on a shortended URL the service should redirect them to the original link.
  • Users should be able to customize the shortened URL or the system should generate one on their behalf.
  • Links with no clicks should expire after a certain amount of time.
  • The system will expose a REST API

The system should also satisfy the principals of system design (i.e. availability, performance, reliability, manageability, and cost). There will always be trade-offs depending on the goals of the system but each area should be taken into account.

Bonus: The system should produce analytics about URL usage.


3. Designing for scale

If the system is popular we should expect it to be light on writes and heavy on reads. Let's assume 100:1 ratio between read and write
